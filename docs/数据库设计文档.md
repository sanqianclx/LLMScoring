# 数据库设计文档

本文档说明本项目当前 MySQL 数据库的设计方案，包含数据库总体说明、表结构设计、表间关系、索引设计以及后续优化建议。

## 1. 数据库概述

### 1.1 数据库名称
- `llm_scoring`

### 1.2 数据库用途
本数据库用于支撑“基于 LLM 的自动评分系统”的核心业务，主要保存以下数据：
- 教师账号信息
- 课程信息
- 试卷信息
- 题目信息
- 题目评分点信息
- 学生提交记录
- 学生答案内容
- 自动评分结果
- 教师审核后的最终评分结果
- 评分命中点与失分点明细

### 1.3 字符集设置
在初始化脚本中使用：
- 字符集：`utf8mb4`
- 排序规则：`utf8mb4_unicode_ci`

该配置可以较好支持中文、英文及常见符号存储。

## 2. 设计目标

本数据库设计遵循以下原则：
1. 满足教师建课、建卷、分享、审核和学生答题、查分的完整业务流程。
2. 尽量采用规范化多表设计，避免把复杂业务结构直接序列化到单字段中。
3. 保留自动评分和人工审核两个阶段的数据，便于结果追踪和后续扩展。
4. 通过主键、外键和索引支撑常见查询场景。
5. 与当前 JPA 实体结构保持一致，便于代码与数据库一一对应。

## 3. E-R 关系概览

系统核心实体关系如下：

- 一个教师可以拥有多个课程。
- 一个教师可以拥有多张试卷。
- 一个课程下可以有多张试卷。
- 一张试卷下可以有多道题目。
- 一道题目下可以有多个评分点。
- 一张试卷可以对应多个学生提交。
- 一次学生提交可以包含多个答案。
- 一次学生提交可以包含多个单题评分记录。
- 一条单题评分记录可以包含多个评分点明细。

可以概括为：
- `teachers 1 - n courses`
- `teachers 1 - n papers`
- `courses 1 - n papers`
- `papers 1 - n questions`
- `questions 1 - n question_scoring_points`
- `papers 1 - n submissions`
- `submissions 1 - n submission_answers`
- `submissions 1 - n submission_question_scores`
- `submission_question_scores 1 - n submission_score_points`

## 4. 表结构设计

### 4.1 `teachers`
作用：保存教师账号与基础资料。

字段说明：
- `id`：主键，教师 ID，`VARCHAR(36)`，使用 UUID。
- `name`：教师姓名。
- `username`：登录用户名，唯一。
- `password`：教师密码，当前版本为明文存储。
- `school`：所属学校。
- `taught_course`：任教课程描述，当前以字符串形式保存。
- `created_at`：创建时间。

主键：
- `id`

唯一约束：
- `username`

设计说明：
- 该表是教师端所有业务数据的根节点。
- 当前版本未拆分角色表与权限表，适用于课程项目阶段。

### 4.2 `courses`
作用：保存课程信息。

字段说明：
- `id`：主键，课程 ID。
- `teacher_id`：所属教师 ID。
- `name`：课程名称。
- `description`：课程描述。
- `created_at`：课程创建时间。

主键：
- `id`

外键：
- `teacher_id -> teachers.id`

索引：
- `idx_courses_teacher_id (teacher_id)`

设计说明：
- 课程作为教师管理试卷的组织维度。
- 当前一个课程只归属一个教师，适合当前单教师管理模式。

### 4.3 `papers`
作用：保存试卷主信息。

字段说明：
- `id`：主键，试卷 ID。
- `teacher_id`：所属教师 ID。
- `course_id`：所属课程 ID。
- `title`：试卷标题。
- `description`：试卷说明。
- `share_code`：试卷分享码，唯一。
- `active`：试卷是否开放。
- `created_at`：创建时间。
- `updated_at`：更新时间。

主键：
- `id`

外键：
- `teacher_id -> teachers.id`
- `course_id -> courses.id`

唯一约束：
- `share_code`

索引：
- `idx_papers_teacher_id (teacher_id)`
- `idx_papers_course_id (course_id)`

设计说明：
- 教师端通过试卷完成题目录入与分享。
- `share_code` 是学生访问试卷的核心入口，因此设置唯一约束。

### 4.4 `questions`
作用：保存试卷下的题目。

字段说明：
- `id`：主键，题目 ID。
- `paper_id`：所属试卷 ID。
- `type`：题型，当前为 `FILL_BLANK` 或 `SHORT_ANSWER`。
- `text`：题目内容。
- `reference_answer`：参考答案。
- `max_score`：题目满分。
- `sort_order`：题目顺序。

主键：
- `id`

外键：
- `paper_id -> papers.id`

索引：
- `idx_questions_paper_id (paper_id)`

设计说明：
- 将题目从试卷主表中拆出，便于支持多题结构。
- `sort_order` 用于保证前端展示和评分时的题目顺序一致。

### 4.5 `question_scoring_points`
作用：保存题目的评分点配置。

字段说明：
- `id`：主键，评分点 ID。
- `question_id`：所属题目 ID。
- `keyword`：关键词或别名表达式。
- `score`：该评分点的分值。
- `description`：评分点说明。
- `sort_order`：评分点顺序。

主键：
- `id`

外键：
- `question_id -> questions.id`

索引：
- `idx_scoring_points_question_id (question_id)`

设计说明：
- 评分点从题目表中拆分出来，便于一个题目配置多个得分点。
- `keyword` 支持多个别名写法，当前后端通过分隔符做拆分匹配。

### 4.6 `submissions`
作用：保存学生提交主记录。

字段说明：
- `id`：主键，提交记录 ID。
- `paper_id`：所属试卷 ID。
- `teacher_id`：归属教师 ID。
- `course_id`：归属课程 ID。
- `share_code`：提交时使用的分享码。
- `student_id`：学生学号。
- `student_name`：学生姓名。
- `auto_total`：自动评分总分。
- `final_total`：最终评分总分。
- `overall_feedback`：整体评语。
- `status`：提交状态，当前为 `PENDING_REVIEW` 或 `REVIEWED`。
- `submitted_at`：提交时间。
- `reviewed_at`：审核时间。

主键：
- `id`

外键：
- `paper_id -> papers.id`
- `teacher_id -> teachers.id`
- `course_id -> courses.id`

索引：
- `idx_submissions_teacher_id (teacher_id)`
- `idx_submissions_share_student_time (share_code, student_id, submitted_at)`

设计说明：
- 该表是学生答卷与评分结果的主入口。
- 联合索引用于支持“按分享码 + 学号查询最近一次结果”的业务场景。

### 4.7 `submission_answers`
作用：保存学生每道题的原始答案。

字段说明：
- `id`：主键，答案记录 ID。
- `submission_id`：所属提交记录 ID。
- `question_id`：题目 ID。
- `answer_text`：学生答案内容。
- `sort_order`：题目顺序。

主键：
- `id`

外键：
- `submission_id -> submissions.id`

索引：
- `idx_submission_answers_submission_id (submission_id)`

设计说明：
- 这里没有对 `question_id` 建外键约束到 `questions.id`。
- 这样设计的原因是：即使后续教师修改或删除试卷题目，历史提交记录仍能保留原始作答信息，避免因题目变动导致历史数据失效。

### 4.8 `submission_question_scores`
作用：保存单题评分结果。

字段说明：
- `id`：主键，评分记录 ID。
- `submission_id`：所属提交记录 ID。
- `question_id`：题目 ID。
- `phase`：评分阶段，当前为 `AUTO` 或 `FINAL`。
- `score`：本题得分。
- `max_score`：本题满分。
- `comment`：评语。
- `rationale`：评分依据。
- `overridden`：是否被教师覆盖修改。
- `sort_order`：题目顺序。

主键：
- `id`

外键：
- `submission_id -> submissions.id`

索引：
- `idx_submission_scores_submission_id (submission_id)`

设计说明：
- `phase` 字段用于区分自动评分结果和最终评分结果。
- 同一题目会保留两类评分数据：
  - `AUTO`：系统自动评分
  - `FINAL`：教师审核后的最终评分
- `overridden` 用于标记该题是否被教师人工调整过。

### 4.9 `submission_score_points`
作用：保存单题评分中的命中点和失分点明细。

字段说明：
- `id`：主键，评分点明细 ID。
- `question_score_id`：所属单题评分记录 ID。
- `point_text`：评分点文本。
- `point_kind`：评分点类型，当前为 `MATCHED` 或 `MISSING`。
- `sort_order`：排序字段。

主键：
- `id`

外键：
- `question_score_id -> submission_question_scores.id`

索引：
- `idx_submission_score_points_score_id (question_score_id)`

设计说明：
- 该表把“命中了哪些得分点”“缺失了哪些得分点”拆出来单独存储。
- 便于前端高亮展示命中与失分原因，也为后续做统计分析预留了空间。

## 5. 主外键关系汇总

| 主表 | 子表 | 关系 | 说明 |
| --- | --- | --- | --- |
| `teachers` | `courses` | 1 对多 | 一个教师可创建多个课程 |
| `teachers` | `papers` | 1 对多 | 一个教师可创建多张试卷 |
| `courses` | `papers` | 1 对多 | 一个课程下可包含多张试卷 |
| `papers` | `questions` | 1 对多 | 一张试卷下可有多道题目 |
| `questions` | `question_scoring_points` | 1 对多 | 一道题可有多个评分点 |
| `papers` | `submissions` | 1 对多 | 一张试卷可有多个学生提交 |
| `submissions` | `submission_answers` | 1 对多 | 一次提交可包含多个答案 |
| `submissions` | `submission_question_scores` | 1 对多 | 一次提交可包含多个评分结果 |
| `submission_question_scores` | `submission_score_points` | 1 对多 | 一条评分记录可包含多个评分点明细 |

## 6. 索引设计说明

当前数据库主要索引及其用途如下：

### 6.1 课程相关索引
- `idx_courses_teacher_id`
  作用：支持教师工作台按教师查询课程列表。

### 6.2 试卷相关索引
- `idx_papers_teacher_id`
  作用：支持按教师查询试卷列表。
- `idx_papers_course_id`
  作用：支持按课程查询试卷列表。
- `share_code` 唯一索引
  作用：支持学生按分享码快速访问试卷。

### 6.3 提交相关索引
- `idx_submissions_teacher_id`
  作用：支持教师按自身查看学生提交记录。
- `idx_submissions_share_student_time`
  作用：支持学生按分享码和学号查询最近一次成绩结果。

### 6.4 子表索引
- `idx_questions_paper_id`
- `idx_scoring_points_question_id`
- `idx_submission_answers_submission_id`
- `idx_submission_scores_submission_id`
- `idx_submission_score_points_score_id`

这些索引用于提升级联读取性能，减少按主记录查询子记录时的扫描成本。

## 7. 与 JPA 实体的对应关系

当前数据库表与后端实体一一对应：

- `teachers` 对应 `TeacherEntity`
- `courses` 对应 `CourseEntity`
- `papers` 对应 `PaperEntity`
- `questions` 对应 `QuestionEntity`
- `question_scoring_points` 对应 `ScoringPointEntity`
- `submissions` 对应 `SubmissionEntity`
- `submission_answers` 对应 `SubmissionAnswerEntity`
- `submission_question_scores` 对应 `SubmissionQuestionScoreEntity`
- `submission_score_points` 对应 `SubmissionScorePointEntity`

这意味着：
- 数据库结构和后端持久化代码保持一致。
- JPA 可以直接完成实体与表之间的映射。
- 后续维护数据库结构时，也应同步更新实体类与初始化脚本。

## 8. 设计优点

当前数据库设计的主要优点：
1. 结构规范，试卷、题目、评分点、提交、评分结果均独立拆表。
2. 自动评分与最终评分分离，便于保留评分过程和审核痕迹。
3. 历史提交数据与试卷结构解耦，能降低后续修改试卷时对历史记录的影响。
4. 可以支持教师端、学生端和评分模块的主要业务查询。
5. 适合当前课程项目、毕设项目或小型教学系统原型使用。

## 9. 当前不足与优化方向

### 9.1 当前不足
1. 教师密码仍为明文存储，安全性不足。
2. `taught_course` 仍为字符串字段，没有进一步拆成规范化多表。
3. 学生当前未单独建表，学生身份通过提交记录中的 `student_id` 和 `student_name` 表示。
4. 缺少操作日志表、审核历史表、权限角色表等更完整的工程化设计。
5. 部分分数字段使用 `DOUBLE`，若后续对精度要求更高，可改为 `DECIMAL`。

### 9.2 后续优化建议
1. 增加密码加密存储，例如 BCrypt。
2. 新增 `students` 表，统一维护学生信息。
3. 新增 `teacher_roles`、`permissions` 等权限控制表。
4. 增加 `review_logs` 表，记录每次教师审核修改历史。
5. 对成绩字段改用 `DECIMAL(5,2)` 等更稳定的数值类型。
6. 结合 Flyway 或 Liquibase 做数据库版本管理。
7. 若后续接入真实 LLM，可增加 `llm_score_logs` 表，记录模型输入输出和评分版本。

## 10. 初始化脚本说明

当前数据库初始化相关文件：
- `src/main/resources/schema.sql`
- `database/mysql-init.sql`

说明：
- `schema.sql` 用于 Spring Boot 启动时初始化表结构。
- `mysql-init.sql` 用于手动初始化 MySQL 数据库与表结构。

推荐初始化命令：

```powershell
mysql -uroot -p123456 < D:\idealcode\LLMScoring\database\mysql-init.sql
```

## 11. 总结

本项目数据库采用面向业务流程的规范化设计，已经能够支撑教师建课建卷、学生答题、自动评分、教师审核和成绩发布等完整场景。当前结构适合作为教学项目和毕业设计项目的数据库基础版本，后续可以在安全性、权限体系、日志记录和扩展能力方面继续增强。
