## 3. 测试覆盖范围

### 3.1 评分正确性测试

`HeuristicScoringServiceTest` 使用人工设定的期望分数校验以下场景：

- 标准答案命中
- 别名命中
- 简答题关键点全部覆盖
- 简答题部分覆盖
- 空白答案
- 概念性错误答案
- 未配置评分点时的参考答案相似度评分

这些测试数据集中在 `heuristic-cases.json`，便于后续继续补充更多课程知识点和错误类型。

### 3.2 评分链路稳定性测试

`ScoringServiceTest` 验证以下逻辑：

- LLM 关闭时，系统使用启发式评分
- LLM 开启但配置不完整时，系统按配置回退到启发式评分
- LLM 正常返回时，系统采用 LLM 评分结果
- LLM 调用失败且允许回退时，系统自动降级
- LLM 调用失败且不允许回退时，系统抛出明确错误

### 3.3 业务流程集成测试

`PlatformServiceIntegrationTest` 基于 H2 测试数据库验证：

- 系统启动后能正确加载演示教师和演示试卷
- 学生能够通过分享码获取试卷并提交答案
- 提交后系统生成自动评分、总分和整体评语
- 教师能够人工审核并覆盖单题得分和评语
- 学生查询结果时，审核前后状态变化正确

## 4. 评分准确性验证方法

为了满足“与人工评分对比”的要求，本次采用以下做法：

1. 先手工给出每个测试答案的预期分数。
2. 将预期分数写入 `heuristic-cases.json`。
3. 由自动化测试将系统实际分数与人工预期分数逐条比对。
4. 若后续接入真实 LLM，可沿用同一份测试集继续对 Prompt 和权重做回归验证。

这意味着后续只需要补充更多题目和答案，不需要重写测试框架。

## 5. 评语优化说明

本次对启发式评分评语做了三类优化：

- 从“泛泛提示”改为“可执行建议”，例如提示学生补充条件、过程、结果。
- 让单题评语和整体评语分别承担不同作用。
- 单题评语聚焦该题缺漏，整体评语聚焦整体掌握程度和下一步改进方向。

当前评语风格更适合课程项目演示，也更容易被教师和学生直接理解。

## 6. 运行方式

项目依赖 Java 21。测试命令如下：

```powershell
mvn test
```

如果本机默认 JDK 不是 21，需要先切换 `JAVA_HOME` 再运行。
